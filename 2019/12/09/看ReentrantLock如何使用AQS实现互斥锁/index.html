分析

我们为三个线程设置了三个不同的睡眠时间，最后一个睡眠时间最小。如果三个线程同时开始肯定是最后一个线程（线程2）最先结束。而当我们使用了lock()方法后，就能使三个线程，按先后顺序执行，从而实现了线程的互斥。通过对ReentrantLock的简单的使用后，我们知道我们应该从lock方法为入口分析AQS的互斥实现

锁的获取

通过对代码的跟踪可以看到ReentrantLock的lock()方法最终就是调用了FairSync的lock()方法，而此方法则是调用了AbstractQueuedSynchronizer的acquire方法。

  • arg参数：从注释中了解到，这个值是被传给tryAcquire的，但是没有其他含义，你可以拿它代表你喜欢的任何事物。

    acquire的执行步骤

  • tryAcquire(arg) 首先尝试获取锁，返回true代表获取成功，返回false代表获取失败。

  • 获取锁失败后，就为将当前线程以独占方式加入到一个队列中（EXCLUSIVE中文释义：adj. 独有的；排外的；专一的)。

tryAcquire(arg)方法

在AbstractQueuedSynchronizer中的tryAcquire方法是未实现的，它的实现留给子类来实现。我们刚刚是从FairSync进来的，所以我们回到FairSync方法查看tryAcquire(arg)方法

这里有个很重要的东西state变量，其反应着锁的状态。

if (c == 0)

state：从代码中可以看到当state为0使，就有机会成功的获取到锁，state为0代表着锁是空闲的。获取到锁后，就使用CAS修改state为acquires（也就是lock.lock(arg)方法传进来的参数）并将锁的拥有者设置为当前线程setExclusiveOwnerThread(current);

else if (current == getExclusiveOwnerThread())

当state != 0，也就是锁被占用的时候，就可以看看当前线程是不是等于占用锁的线程。如果等于且c + acquires > 0，那么仍然可以获取锁。这就是锁的重入。

其中有个疑惑就是hasQueuedPredecessors方法是干嘛的？
为啥hasQueuedPredecessors为false才能获取锁？

返回false的条件：

h == t

在AQS中head节点是一个空节点不放线程的，所以当h == t时也就是说队列中没有线程在等待获取锁（换句话说就是没有人跟你竞争锁），那你当然能够直接获取锁。

h.next != null && h.next.thread == Thread.currentThread()

当队列不为空时，我们就要判断队列中第一个节点是不是本线程。如果是本线程则允许获取锁。
